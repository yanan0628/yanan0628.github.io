  • 2).节点6381接受到meet消息后，保存6380节点信息并回复pong消息。
  • 3).之后节点6380定期发送ping消息进行正常的节点通信。

如何保证节点部分失败，集群仍然工作:

如果集群槽意外丢失，可能会造成集群不可用，所有请求都不可用，在我们运维redis集群过程中，肯定会遇到机器宕机、进程kill等等情况，可能会造成槽丢失。这里redis提供一个参数控制：

cluster-require-full-coverage:yes(默认值)/no 集群丢失槽后，是否接受客户端调用

yes:不接受客户端调用
no: 正常数据槽仍然可以接受客户端调用

集群伸缩

Redis集群提供了灵活的节点扩容和收缩方案。在运维期间:
1).避免不了因机器下线、宕机需要对节点迁移;
2) 集群容量扩容由于分片足够大需要做水平扩容;
3).缩容需求 等等
这里主要讲redis原生命令操作。

参考文章：[redis-trib文章]、[cachecloud水平扩容]

集群伸缩的原理: 其实就是变更节点-slot关系的过程，当我们在增加或减少redis节点过程期间需要将部分槽迁移到指定目标节点。

相关流程和命令参考: 集群官方文档 集群相关命令

槽迁移

不论扩容还是缩容，最重要一步就是槽迁移过程:

  • 1).目标节点准备导入槽：

    源节点 > cluster setslot ${slot} importing ${targetNodeid}

  • 2).源节点准备导出槽

    目标节点 > cluster setslot ${slot} migrating ${sourceNodeId}

  • 3).源节点获取slot的键【循环开始】

    目标节点 > cluster getkeysinslot ${slot} ${count}

  • 4).源节点开始迁移slot上的槽的键，直到迁移槽上所有键 【循环结束】

    目标节点 > migrate ${targetHost} ${targetPort} “” 0 ${timeout} keys ${key}

  • 6).通知所有master节点更新 槽的变化。

    每个master节点 > cluster setslot {slot} node {targetNodeId}

实例槽进行迁移

现在redis-cluster集群有四个节点 6380-6383,先讲6383节点(sourceNode)槽slot=0 迁移到 6380节点(targetNode)

开始迁移:

  • 1).目标节点准备导入槽
  • 2).源节点准备导出槽
  • 3,4).循环取出源节点键，migrate到目标节点

集群缩容

收缩集群意味着缩减规模，需要从现有集群中安全下线部分节点。安全下线节点流程:

  • 1).首先需要确定下线节点是否持有槽， 如果是需要把槽迁移到其它节点，保证节点下线后整个集群槽节点映射的完整性。
  • 2).当节点不再负责槽或者本身是从节点时，就可以通知集群内其它节点忘记下线节点。

  • 3).当集群内所有的节点忘记该节点后，可以正常关闭节点从而完成安全下线。

在集群伸缩这一节，主要说道槽迁移过程的步骤较为复杂，集群扩容和缩容也是对于槽的一个迁移过程，可以使用redis-trib.rb工具或代码迁移去解决，手工迁移较为麻烦。
